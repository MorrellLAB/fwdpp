1 # Tutorial 0: Data types in fwdpp
2 
3 This document introduces the fundamental data types for simulations implemented using __fwdpp__.
4 
5 The relevant header file defining these types is
6 
7 ~~~{.cpp}
8 #include <fwdpp/forward_types.hpp>
9 ~~~
10 
11 This header will be included when you include the main library header via
12 
13 ~~~{.cpp}
14 #include <fwdpp/diploid.hh>
15 ~~~
16 
17 __Note:__ this document only covers single-population, individual-based simulations in detail.
18 
19 ## Mutation types
20 
21 The most basic type determining the behavior of a simulation is a mutation. Any __fwdpp__-based simulation must define a mutation type that publicy inherits from KTfwd::mutation_base. This base class contains several public data types:
22 
23 ~~~{.cpp}
24 //The mutation position
25 mutable double pos;
26 //The number of occurrences in the population
27 unsigned n;
28 //Does this mutation affect fitness/trait value?
29 bool neutral;
30 //This is used and updated internally by other functions w/in the library
31 bool checked;
32 ~~~
33 
34 This type is usable in a simulation, but not in an interesting simulation, as there is no selection coefficient, etc. In order to add things to a mutation type, you extend KTfwd::mutation_base via public inheritance. You may find an example of that in the next tutorial (@ref md_md_policies) and in the following types provided by the library:
35 
36 * KTfwd::mutation
37 * KTfwd::popgenmut
38 
39 ### Notes
40 
41 * Remember to _properly_ initialize the base class from your derived mutation classes!
42 * All fitness models provided by the library require a mutation type to contain a double called \f$s\f$ representing the selection coefficient (or effect size). For example, see KTfwd::site_dependent_fitness. Hoewever, if you write your own fitness models, then your mutations can contain whatever they want.

44 ## Mutation lists
45 
46 Mutations must be stored in double-linked lists. In the C++ standard library, use std::list:
47 
48 ~~~{.cpp}
49 #include <list>
50 #include <fwdpp/diploid.hh>
51 
52 std::list< KTfwd::mutation > mlist;
53 ~~~
54 
55 The standard list type works well, and simulations will be pretty fast. However, we can do better. In any forward simulations, mutations enter the population and then quickly exit because they are lost due to drift and/or selection. Situations where small objects are coming and going rapidly screams for a memory pool allocation model, rather than the standard allocator that the above code block is implying:
56 
57 ~~~{.cpp}
58 #include <list>
59 #include <fwdpp/diploid.hh>
60 
61 //This is equivalent to the above block,
62 //which relies on the default allocator
63 //being the standard one.
64 std::list< KTfwd::mutation, std::allocator<KTfwd::mutation> > mlist;
65 ~~~
66 
67 The [boost](http://www.boost.org) C++ libraries provide nicely-implemented memory pool objects:
68 
69 ~~~{.cpp}
70 #include <boost/container/list.hpp>
71 #include <boost/pool/pool_alloc.hpp>
72 #include <fwdpp/diploid.hh>
73 
74 //We see that typedefs lead to improved sanity:
75 using mutation_t = KTfwd::mutation;
76 using mutation_t_allocator = boost::pool_allocator<mutation_t>;
77 boost::container::list< mutation_t, mutation_t_allocator > mlist;
78 ~~~
79 
80 For simulations with large mutation and/or recombination rates, using the boost types can result is substantially faster simulations. It is well worth it to have boost available on your systems!
81 
82 Note that you can use the boost allocators with the standard containers, etc.
83 
84 Further note that I have done _zero_ experimentation with parameters to the pool_allocator template, which itself can be tweaked in various ways.

86 ## Gamete types
87 
88 The basic gamete type is KTfwd::gamete_base, which is a template type. Fundamentally, gametes contain vectors if _iterators_ derived from mutation lists. There is one such vector for "neutral" variants, another for "selected" variants.
89 
90 In order to instantiate a gamete template, you must provide the mutation type at a minimum:
91 
92 ~~~{.cpp}
93 #include <list>
94 #include <fwdpp/diploid.hh>
95 
96 using mutation_t = KTfwd::mutation;
97 std::list< mutation_t > mlist;
98 //This defaults to assuming that std::list<mutation_t> is the mutation list type
99 using gamete_t = KTfwd::gamete_base<mutation_t>;
100 ~~~
101 
102 If your mutation list type is not std::list, then you must provide an additional template parameter:
103 
104 ~~~{.cpp}
105 #include <boost/container/list.hpp>
106 #include <boost/pool/pool_alloc.hpp>
107 #include <fwdpp/diploid.hh>
108 
109 //We see that typedefs lead to improved sanity:
110 using mutation_t = KTfwd::mutation;
111 using mutation_t_allocator = boost::pool_allocator<mutation_t>;
112 using mlist_t = boost::container::list< mutation_t, mutation_t_allocator >;
113 mlist_t mlist;
114 using gamete_t = KTfwd::gamete_base<mutation_t,mlist_t>;
115 ~~~
116 
117 That's basically it for gametes.
118 
119 ### Notes
120 
121 * The interface to Ktfwd::gamete_base is a little funny. Ideally, you should be able to provide only the mutation list type, _e.g._,
122 
123 ~~~{.cpp}
124 using gamete_t = KTfwd::gamete_base<mutation_t,mlist_t>;
125 ~~~
126 
127 The gamete_t should be able to deduce the mutation_t from the mlist_t::value_type. That's all true, but I first wrote the gamete type based around the idea of the mutation and the default std::list<mutation_t>. Changing it now would break all existing simulations using the library.
128 
129 * As of __fwdpp__ 0.3.0, KTfwd::gamete_base takes a third template argument, which currently defaults to KTfwd::tags::standard_gamete. This tag has no effect on anything, and the library remains source-compatible with previous versions. This "tag" is a placeholder for potential future library features making use of tag-dispatch methods for implementing more complex models.

132

135 ## Diploids
136 
137 In fwdpp, a diploid is a pair of iterator derived from a gamete list, _e.g._,
138 
139 ~~~{.cpp}
140 using glist_t = std::list<gamete_t>;
141 using diploid_t = std::pair< glist_t::iterator, glist_t::iterator >;
142 ~~~
143 
144 Then, the entire population is a vector of diploids:
145 
146 ~~~{.cpp}
147 std::vector< diploid_t > diploids;
148 ~~~
