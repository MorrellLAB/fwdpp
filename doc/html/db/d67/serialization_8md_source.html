10 ## The basic functions
11 
12 For single-deme simulations, the library provides the following functions:
13 
14 * KTfwd::write_binary_pop
15 * KTfwd::read_binary_pop
16 
17 These functions write and read simulated populations in a binary (_e.g._ not human-readable) format.
18 
19 For multi-deme simulations, we have:
20 
21 * KTfwd::write_binary_metapop
22 * KTfwd::read_binary_metapop
23 
24 As with the rest of the library, these functions are implemented using a combination of templates + function overloading.
25 
26 Currently, these functions support the following types of simulations:
27 
28 * single-locus, single-deme
29 * single-locus, multi-deme
30 * multi-locus, single-deme
31 
32 The type requirements for these functions are:
33 
34 * Containers of objects used in fwdpp-based simulations (containers of gametes, mutations, diploids, etc.)
35 * Output streams compatible with the public interface of [std::ostream](http://www.cplusplus.com/reference/ostream/ostream/)
36 * Input streams compatible with the public interfact of [std::istream](http://www.cplusplus.com/reference/istream/istream/)
37 * Input via the gzFile type defined by [zlib](http://zlib.net)
38 
39 ## Writing and reading mutations
40 
41 You need to tell these functions how to read/write mutation objects. Specifically, you need to:
42 
43 * Define a mutation write function, which takes a mutation object and an output stream type as an object.
44 * Define a mutation read function, taking an input stream type as an object
45 
46 Let's look at an example. For this definition of a mutation type:
47 
48 ~~~{.cpp}
49 struct mutation_with_age : public KTfwd::mutation_base
50 {
51  double s,h;
52  unsigned g; //generation in which mutation arose
53  mutation_with_age(const unsigned & __o,const double & position, const unsigned & count,
54  const double & __s, const double __h,
55  const bool & isneutral = true)
56  : KTfwd::mutation_base(position,count,isneutral),s(__s),h(__h),g(__o)
57  {
58  }
59 };
60 ~~~
61 
62 This function object works as a writer:
63 
64 ~~~{.cpp}
65 //function object to write mutation data in binary format
66 struct mwriter
67 {
68  typedef void result_type;
69  result_type operator()( const mutation_with_age & m, std::ostream & buffer ) const
70  {
71  unsigned u = m.n;
72  buffer.write( reinterpret_cast< char * >(&u),sizeof(unsigned) );
73  u = m.g;
74  buffer.write( reinterpret_cast< char * >(&u),sizeof(unsigned) );
75  bool b = m.neutral;
76  buffer.write( reinterpret_cast< char * >(&b),sizeof(bool) );
77  double d = m.pos;
78  buffer.write( reinterpret_cast< char * >(&d),sizeof(double) );
79  d = m.s;
80  buffer.write( reinterpret_cast< char * >(&d),sizeof(double) );
81  d = m.h;
82  buffer.write( reinterpret_cast< char * >(&d),sizeof(double) );
83  }
84 };
85 ~~~
86 
87 And this function object works as a reader:
88 
89 ~~~{.cpp}
90 //function object to read mutation data in binary format
91 struct mreader
92 {
93  typedef mutation_with_age result_type;
94  result_type operator()( std::istream & in ) const
95  {
96  unsigned n;
97  in.read( reinterpret_cast< char * >(&n),sizeof(unsigned) );
98  unsigned g;
99  in.read( reinterpret_cast< char * >(&g),sizeof(unsigned) );
100  bool neut;
101  in.read( reinterpret_cast< char * >(&neut),sizeof(bool) );
102  double pos;
103  in.read( reinterpret_cast< char * >(&pos),sizeof(double) );
104  double s;
105  in.read( reinterpret_cast< char * >(&s),sizeof(double) );
106  double h;
107  in.read( reinterpret_cast< char * >(&h),sizeof(double) );
108  return result_type(g,pos,n,s,h,neut);
109  }
110 };
111 ~~~

115 ## Reading and writing diploids.
116 
117 Version 0.3.1 allowed the use of custom diploid genotype objects (@ref md_md_customdip). Such objects can be serialized with no additional effort. _However, doing so will result in custom data associated with these custom types not being serialized._
118 
119 In order to solve this, define your own custom structures for writing and reading:
120 
121 ~~~{.cpp}
122 struct diploid_writer {
123  using result_type = void;
124  template<typename dip_itr, typename streamtype >
125  inline result_type( dip_itr i, streamtype & o ) const
126  {
127  //Do the right thing here.
128  }
129 };
130 ~~~
131 
132 The arguments should be an iterator pointing to one of your custom diploids, and a reference to an output stream (including a gzFile!). A function object of the same form should also be written to read the data back in.
133 
134 These custom serialization objects may be passed as the last arguments to the functions discussed in the next section. By default, they are implicitly passed KTfwd::diploidIOplaceholder, which is essentially an empty object.

136 ## In-memory copying
137 
138 It may be desirable to be able to restore a population from a previous state. For example, you may wish to repeatedly introduce a mutation at a position, and then simulate until it is fixed. For replicates where it is lost, you will restore the population to the state it was in when you introduced the mutation. You may do this using KTfwd::write_binary_pop or KTfwd::write_binary_metapop, and write the population to an in-memory buffer such as [std::ostringstream](http://www.cplusplus.com/reference/sstream/ostringstream/). You may restore the population by reading the data from a [std::istringstream](http://www.cplusplus.com/reference/sstream/istringstream/).
139 
140 Here is some pseudocode for an individual-based simulation:
141 
142 ~~~{.cpp}
143 //run a simulation...
144 
145 //write it to a buffer:
146 std::ostringstream buffer;
147 KTfwd::write_binary_pop(&gametes,&mutations,&diploids,std::bind(mwriter(),std::placeholders::_1,std::placeholders::_2),buffer);
148 
149 //Now, if you want to restore it:
150 std::istringstream inbuffer(buffer.str());
151 decltype(mutations) mutations2;
152 decltype(gametes) gametes2;
153 decltype(diploids) diploids2;
154 KTfwd::read_binary_pop(&gametes2,
155  &mutations2,
156  &diploids2,
157  std::bind(mreader(),std::placeholders::_1),
158  inbuffer);
159 ~~~
160 
161 This approach works because KTfwd::write_binary_pop and KTfwd::write_binary_metapop perform "deep copies" of the data, allowing the complete restoration of all the pointers, etc., stored in the containers.

163 ## Writing to files
164 
165 You may use the same functions to write/read to from files. You may use and sort of [std::ostream](http://www.cplusplus.com/reference/ostream/ostream/)-compatible object for output. You may use either a [std::istream](http://www.cplusplus.com/reference/istream/istream/)-compatible object for input or a gzFile object from [zlib](http://zlib.net).

167 ## Managing output from multiple independent processes
168 
169 In practice, we often implement simulations with the idea that one replicate will be run per process, and that we will spread a large number of processes across our cluster. Typically, we collect the output of these simulations into a single file. To do this, we implement POSIX-style advisory file locking. See [here](https://github.com/molpopgen/BigDataFormats) for a discussion of how to do that using the low-level C functions found in <fcntl.h>. A more modern C++-based approach would be to use the [boost](http://www.boost.org) synchronization library. See [here](https://gist.github.com/molpopgen/651e4ac81253f34364f7) for simple examples.
170 
171 ### Why not use boost serialization?
172 
173 Cliff's notes version: this approach would work well for a single process writing lots of replicates to one file. However, the archives that this library creates do not support appending on to an existing library. This limitation is unfortunate, as being able to use this library would simplify some of the implementation.
